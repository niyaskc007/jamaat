# Member Portal — Admin Runbook

This document is for administrators. It covers everything you need to onboard members onto the self-service portal: turning the system on, creating logins, sharing temporary passwords, and reading the audit trail.

---

## What is the member portal?

A self-service area at **`/portal/me`** where each member can:

- View their profile, contributions, commitments, Qarzan Hasana loans, event registrations
- Submit a new commitment or QH application (which then enters the standard approval workflow)
- Endorse or decline guarantor requests addressed to them
- See their own login history
- Change their password

It runs alongside the existing operator app under a separate layout. Members never see admin / accounting / approver screens — even with a stolen JWT, the `portal.access` policy gate blocks them.

---

## How members get an account

Every member with a record in the system already has a corresponding `ApplicationUser`:

- The auto-provisioning runs on **every API startup** (the database seeder calls `IMemberLoginProvisioningService.BackfillTenantAsync`).
- Newly-created members (via `POST /api/v1/members` or the bulk Excel import) get provisioned **immediately** as part of that flow.
- Each provisioned user starts with:
  - `UserName` = the ITS number
  - `Email` = whatever's on the Member record (may be null)
  - `IsLoginAllowed` = **false** (they cannot log in until you explicitly enable them)
  - `MustChangePassword` = **true**
  - A cryptographically random **temporary password** generated by `TemporaryPasswordService`
  - Default role: **Member** (only `portal.*` permissions)

This gate is intentional — it prevents an Excel import from immediately exposing thousands of accounts before anyone is ready to handle support questions.

## Onboarding flow (one member or many)

### One member at a time

1. Sign in as admin → **Administration → Users & Roles → Users**.
2. Find the member (search by name or ITS).
3. Click **Manage** to open the drawer.
4. Open the **Temp password** tab. The current temporary password is displayed in a copy-able box with the expiry date.
5. Click **Copy** and share it with the member out-of-band (in person, secure phone, etc.) — *or* let the welcome notification send it for you (see "Notifications" below).
6. Back in the user list, tick the member's row and click **Enable login** in the bulk-action bar.
7. The member can now sign in at `/login`.

### Bulk onboarding (after Excel import)

1. **Administration → Users & Roles → Users**.
2. Tick all the rows you want to enable (or use the column header checkbox to select the whole page).
3. Click **Enable login** in the bulk-action bar.
4. If a notification provider is configured (SMS / WhatsApp / Email), each enabled user receives a welcome message containing their portal URL + temporary password. The toast confirms how many users were enabled and how many were notified.

## How members sign in (the first time)

1. Member opens `https://your-host/login`.
2. Types their **ITS number** OR their **email address** in the email field, plus the temporary password.
3. Server detects `MustChangePassword=true`, returns `PasswordChangeRequiredResponse` instead of a JWT.
4. Frontend redirects to `/change-password` with the identifier + temp password pre-filled.
5. Member chooses a permanent password (min 8 chars) and submits.
6. Server clears the temp password, stamps `LastPasswordChangedAtUtc`, issues a real JWT.
7. Member lands on `/portal/me` (because they only carry `portal.*` permissions).

If the member fails to change their password before the temp expires (default 7 days, configurable via `TemporaryPassword:ExpiryDays`), login is blocked and admin must re-issue.

---

## Re-issuing a temporary password

Member lost the temp password / it expired / they're locked out:

1. **Users → Manage → Temp password**.
2. Click **Re-issue temporary password**. The previous one is invalidated immediately.
3. Copy the new plaintext from the box and share it.
4. If notifications are configured, a *Temp password issued* message goes to the member automatically.

## Disabling a member's access

Two distinct toggles:

- **Login allowed** (admin-controlled) — Phase B1 toggle. Set false to block sign-in without affecting their data record.
- **Active** (directory) — flips the user inactive in the global directory; also blocks login.

Use **Login allowed = false** when you want to suspend portal access without flagging the member as inactive in the directory.

## Login history & geolocation

Every login attempt — success or failure — is recorded in the `LoginAttempts` table. View them at:

- **Users tab → Login history** sub-tab — tenant-wide, most recent 200.
- **User Manage drawer → Login history** sub-tab — that user's last 50 attempts.

Each row shows when, outcome, IP address, country (and city if you've installed the City DB), device/browser fingerprint. See [`docs/admin/geolocation-maxmind.md`](geolocation-maxmind.md) for installing the offline MaxMind database.

## Notifications & integrations

The portal pipes welcome / temp-password / approval messages through a pluggable channel-routing layer. Configure at **Administration → Integrations**.

**Channel selection priority** (most-direct wins):

1. WhatsApp (if `WhatsApp:Provider` is set AND member has `PhoneE164` on file)
2. SMS (if `Sms:Provider` is set AND member has `PhoneE164`)
3. Email (if `Notifications:Smtp` is configured AND member has email)
4. Log-only (audit row written; no outbound delivery)

**SMS providers shipped:**

| Provider | Region | Approx UAE rate |
|---|---|---|
| Twilio | Global | $0.04+/SMS |
| Unifonic | UAE-based | $0.02-0.03/SMS |
| Infobip | Global, strong MENA | $0.025-0.03/SMS |

Pick one in `Sms:Provider` and supply credentials in the matching `Sms:Twilio*` / `Sms:Unifonic*` / `Sms:Infobip*` keys. See [`docs/admin/integrations.md`](integrations.md).

**WhatsApp:** Twilio Business API only out-of-the-box. Sandbox works for testing without template approval; production needs your templates approved by Twilio.

Every notification — sent or skipped — is recorded in `NotificationLogs` so you have a complete audit trail even when delivery is in log-only mode.

## Permissions reference

The seeded **Member** role carries:

- `portal.access` — gate on `/portal/me`
- `portal.contributions.view.own` — own receipts list
- `portal.commitments.view.own` + `portal.commitments.create.own`
- `portal.qh.view.own` + `portal.qh.request` + `portal.qh.endorse_guarantor`
- `portal.events.view` + `portal.events.register`
- `portal.login_history.view.own`
- `member.self.update` (existing — used by the change-request queue)
- `member.wealth.view` (own data only)

Every controller endpoint under `PortalMeController` filters by the JWT subject's MemberId, so even if you accidentally grant a member another permission, they still cannot read or mutate someone else's data through the portal.

## Configuration knobs

```json
{
  "TemporaryPassword": {
    "ExpiryDays": 7,
    "Length": 12
  },
  "Portal": {
    "PublicUrl": "https://your-host"
  },
  "Seed": {
    "BackfillMemberLogins": true
  },
  "Geolocation": {
    "Provider": "MaxMind",
    "MaxMindDatabasePath": "Maxmind",
    "CacheMinutes": 60
  },
  "Sms": {
    "Provider": "Unifonic",
    "FromNumber": "+9715xxxxxxx"
  },
  "WhatsApp": {
    "Provider": "Twilio",
    "FromNumber": "+9715xxxxxxx"
  }
}
```

| Key | Default | Effect |
|---|---|---|
| `TemporaryPassword:ExpiryDays` | 7 | How long a temp password remains usable from issue. |
| `TemporaryPassword:Length` | 12 | Generated temp password length. |
| `Portal:PublicUrl` | `http://localhost:5173` | Used in welcome notification body. Set to your real host. |
| `Seed:BackfillMemberLogins` | true | Set false to skip the startup backfill of member logins. |
| `Geolocation:Provider` | MaxMind | Active geolocation provider. |
| `Sms:Provider` | (empty) | Empty = SMS disabled. |
| `WhatsApp:Provider` | (empty) | Empty = WhatsApp disabled. |

## Troubleshooting

**Q: Member says "Your account is not enabled for self-service login"**
A: Their `IsLoginAllowed` is false. Tick the row, click **Enable login**.

**Q: Member says "Your temporary password has expired"**
A: Open their drawer → Temp password tab → **Re-issue**. Share the new plaintext.

**Q: Welcome message didn't arrive.**
A: Check **Administration → Notifications**. Every notification — delivered, skipped, or failed — has an audit row with the failure reason. Common causes: SMS provider not configured (status: log-only), member has no phone on file (status: skipped), Twilio credentials wrong (status: failed with 401 details).

**Q: Login history shows no country / city.**
A: MaxMind database not loaded yet. See [`docs/admin/geolocation-maxmind.md`](geolocation-maxmind.md) — upload via Integrations → Geolocation. Loopback / private-range IPs are intentionally excluded.

**Q: Member uses email or ITS for username — which works?**
A: Both. The login flow tries email → username → 8-digit ITS in that order.
